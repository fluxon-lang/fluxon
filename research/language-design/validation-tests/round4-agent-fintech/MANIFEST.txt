FINTECH BACKEND FILES (fintech/haiku/)

schema.fx (81 lines)
  Purpose: Database schema definition
  Audience: Database architects, DBA
  Contains:
    - accounts table (id, owner, currency, type, status, balance_cents)
    - ledger_entries table (immutable, double-entry)
    - transactions table (high-level movements)
    - transfers table (account-to-account)
    - payment_methods table
    - audit_log table (immutable compliance log)
    - reconciliation_log table
    - users table (foreign key references)
  Dependencies: None (pure schema)

accounts.fx (198 lines)
  Purpose: Account management
  Audience: Backend engineers
  Exports:
    - create_account(owner, currency, type) → account
    - get_account(id) → account with fresh balance
    - get_balance(id) → int cents
    - list_accounts(owner_id) → [account]
    - update_account_status(id, status) → account
    - deposit(id, amount_cents) → transaction
    - withdraw(id, amount_cents) → transaction
  Dependencies: db, json, time
  Notes: Uses db.tx for atomicity, includes audit logging

transfers.fx (194 lines) ⭐ MOST COMPLEX
  Purpose: Core transfer logic
  Audience: Backend engineers, money processing
  Exports:
    - transfer(from_id, to_id, amount, currency, idempotency_key) → result
    - reverse_transfer(id) → reversal
    - get_transfer(id) → transfer record
    - list_transfers(account_id) → [transfer]
  Key Features:
    - Atomic: all-or-nothing with full rollback
    - Idempotent: same key = same result
    - Fraudulent transfer detection
    - Double-entry ledger creation (debit + credit)
    - Currency validation
    - Audit trail logging
  Dependencies: db, json, accounts, fraud, idempotency
  Critical Assumptions:
    - db.tx provides SERIALIZABLE isolation
    - db.ins raises error on constraint violation
    - Errors propagate (not silenced)

fraud.fx (91 lines)
  Purpose: Fraud detection and risk scoring
  Audience: Risk/compliance team, fraud analysts
  Exports:
    - check_transfer(from, to, amount, currency) → {blocked, reason, score}
    - get_daily_spent(account_id) → cents
    - get_fraud_score(from, to, amount) → 0..1
    - get_reasons(from, to, amount) → [str]
  Rules:
    1. Daily transfer limit: $1,000 per account (100,000 cents)
    2. Large transfer: >50% of limit → risk += 0.75
    3. New destination: never transferred before → risk += 0.2
    4. Block: score > 0.95
  Dependencies: db, time

idempotency.fx (60 lines)
  Purpose: Prevent double-charging on retried requests
  Audience: Backend engineers, payment specialists
  Exports:
    - check_and_lock(key) → cached result or nil
    - store_result(key, transfer_id, txn_id) → nil
    - create_schema() → nil
  Implementation:
    - In-memory cache (idempotency_cache) for speed
    - DB table (idempotency_log) for persistence
    - UNIQUE constraint on idempotency_key
  Critical Assumption:
    - db.ins raises error on constraint violation (second insert same key)

ai_features.fx (111 lines)
  Purpose: AI-powered explanations and fraud scoring
  Audience: Compliance, risk management, customer support
  Exports:
    - explain_transaction(txn_id) → explanation with fraud score
    - score_transaction_for_fraud(txn_id) → {score, risk_factors}
  Features:
    - Natural language explanation (ai.ask)
    - Structured fraud scoring (ai.json)
    - Risk factors with reasoning
    - Confidence scores
  Dependencies: ai, json, fraud, db

reconciliation.fx (102 lines)
  Purpose: Daily verification of ledger invariants
  Audience: Compliance, data integrity, ops
  Exports:
    - reconcile_all() → reconciliation record
    - get_discrepancies(since_days) → [discrepancies]
    - fix_balance(id) → corrected account
  Implementation:
    - Runs daily at 00:00 UTC (via cron)
    - Verifies: account.balance_cents = sum(ledger_entries) ∀ account
    - Logs all discrepancies with before/after JSON
    - Emergency fix for verified discrepancies
  Dependencies: db, json, time

cron_jobs.fx (18 lines)
  Purpose: Background job scheduling
  Audience: DevOps, system design
  Exports:
    - setup_cron_jobs() → nil
  Schedules:
    - Daily reconciliation at 00:00 UTC
  Dependencies: cron, reconciliation

CORRECTNESS ASSESSMENT
================================================================================

CRITICAL SUCCESS FACTORS (must be true):
  1. db.tx provides SERIALIZABLE isolation
     - Without this: racing transfers can double-spend
     - Can test with: concurrent transfer race test

  2. db.ins raises error on constraint violation
     - Without this: idempotency keys can duplicate
     - Can test with: concurrent same-key insert test

  3. str.int raises error on parse failure
     - Without this: transfers silently go to account 0
     - Can test with: POST with amount="invalid"

  4. Errors propagate upward
     - Without this: partial failures corrupt data
     - Can test with: kill transfer mid-execution

  5. JSON serializes all types
     - Without this: audit log becomes unreliable
     - Can test with: verify audit_log for all field types

RISK PROFILE:
  If all assumptions hold → 95% correct
  If any assumption fails → 30-50% correct (critical bugs)

Highest-risk areas:
  1. Concurrent transfers (double-spend risk)
  2. Idempotency (duplicate charge risk)
  3. Balance consistency (invariant violation)
  4. Type conversions (silent corruption)
