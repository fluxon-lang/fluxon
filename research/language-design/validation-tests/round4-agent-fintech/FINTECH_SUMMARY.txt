ARCHITECTURE OVERVIEW
================================================================================

CORE PRINCIPLE: Double-Entry Accounting
  Every money movement → two balanced ledger entries (debit + credit)
  Invariant: sum(credits) = sum(debits) ∀ account
  Verified daily by reconciliation

MONEY REPRESENTATION: Integer Cents Only
  No floats (IEEE 754 rounding errors)
  All amounts: balance_cents, amount_cents (integer minor units)
  Assumption: int64 safe up to ~$92M per account

ATOMICITY: db.tx (Database Transactions)
  All money-moving ops in single transaction
  All-or-nothing: commit success OR full rollback
  Prevents orphaned ledger entries, torn reads

IDEMPOTENCY: Prevent Double-Charging
  Every transfer takes idempotency_key (UUID)
  Same key twice = return original result (cached)
  Solves: network retry → duplicate charge

AUDIT TRAIL: Immutable Log
  Every state change recorded: actor, action, entity, before/after JSON
  Non-deletable: compliance + debugging
  Detects: unauthorized changes, bugs, data corruption

FRAUD DETECTION: Rules-Based + AI
  Daily transfer limit: $1,000 per account
  Large transfer flags: >50% of limit
  New destination detection: never transferred to before
  AI scoring: LLM analyzes risk with custom reasoning
  Risk score: 0..1, blocks if > 0.95

RECONCILIATION: Daily Verification
  Runs at 00:00 UTC via cron
  Compares: account.balance_cents vs sum(ledger_entries)
  Detects: data corruption, race conditions, bugs
  Emergency fix: restore balance from ledger

SPEC GAPS IDENTIFIED (13 Critical/Major)
================================================================================

1. NO DECIMAL TYPE [CRITICAL]
   Problem: int + flt only. Floats unsafe for money. Integers can overflow.
   Impact: Silent integer overflow at int64 max
   Workaround: Use int cents, manually cap at int64 max
   Risk: HIGH if amounts exceed 64-bit range

2. ATOMICITY GUARANTEES UNSPECIFIED [CRITICAL]
   Problem: Spec shows db.tx syntax but doesn't document isolation level
   Impact: Racing transfers could double-spend if tx < SERIALIZABLE
   Workaround: Assume SERIALIZABLE (Postgres default)
   Risk: CRITICAL if tx isolation is READ_COMMITTED or weaker

3. NO ROW-LEVEL LOCKING [CRITICAL]
   Problem: No SELECT FOR UPDATE, LOCK TABLE, or atomic ops
   Impact: Can't ensure balance check + debit are atomic without tx
   Workaround: Rely entirely on db.tx serialization
   Risk: CRITICAL if tx isolation weak or locking unavailable

4. ATOMIC CHECK-AND-INSERT IMPOSSIBLE [CRITICAL]
   Problem: No INSERT ... ON CONFLICT, no compare-and-swap
   Impact: Idempotency key race: two requests insert same key → duplicate
   Workaround: Add DB UNIQUE constraint, assume violation raises error
   Risk: CRITICAL if db.ins returns nil on constraint violation (silent fail)

5. TYPE CONVERSION UNSPECIFIED [MEDIUM-HIGH]
   Problem: str.int "notanumber" returns ??? (error? nil? 0?)
   Impact: If returns 0, transfers go to account 0 (silent corruption)
   Workaround: Manual guards before using converted values
   Risk: HIGH if conversion is silent

6. INVARIANT CHECKING ABSENT [MEDIUM-HIGH]
   Problem: No assert, invariant, or compile-time checks
   Impact: Data corruption undetected (balance ≠ ledger sum)
   Workaround: Daily reconciliation detects post-facto
   Risk: MEDIUM (detected daily but undetected until then)

7. ERROR MODEL UNCLEAR [MEDIUM]
   Problem: fail vs nil vs error types — no structured error handling
   Impact: Callers can't distinguish "not found" from "validation failed"
   Workaround: Assume fail propagates, return nil for not-found
   Risk: MEDIUM (works if assumptions hold)

8. JSON SCHEMA UNSPECIFIED [MEDIUM]
   Problem: Audit log stores JSON but no schema validation
   Impact: Malformed JSON, missing fields, oversized blobs
   Workaround: Assume all fields JSON-serializable, no size limits
   Risk: MEDIUM (audit trail becomes unreliable)

9. MUTABLE STATE SCOPE UNCLEAR [MEDIUM]
   Problem: Module-level mutable state — per-request or global?
   Impact: Idempotency cache might not work (isolation issue)
   Workaround: Assume global, fall back to DB lookup
   Risk: MEDIUM (cache useless if per-request, but fallback works)

10. TIME FORMATTING UNSPECIFIED [LOW-MEDIUM]
    Problem: time.fmt t "..." syntax not shown (strftime? Go? custom?)
    Impact: Wrong date formatting in reconciliation_log
    Workaround: Assume strftime
    Risk: LOW (only affects date display, core logic unaffected)

11. FLOAT PRECISION [LOW-MEDIUM]
    Problem: flt type has unspecified precision; accumulating floats drift
    Impact: Fraud risk score 0.95000001 ≠ 0.95 (wrong decision)
    Workaround: Manually cap scores at 1.0, use int percentiles
    Risk: LOW (detectable in tests, easily fixed)

12. ERROR PROPAGATION IN LOOPS [MEDIUM]
    Problem: If fail inside each loop, does execution stop? Partial state?
    Impact: Batch operations might partially complete
    Workaround: Avoid ! in loops, explicit error handling
    Risk: MEDIUM (avoided by design)

13. IDEMPOTENCY_KEY COLLISION [MEDIUM]
    Problem: What if user reuses same UUID for different transfers (bug)?
    Impact: Both transfers return same result (wrong!)
    Workaround: Expect UUIDs to be unique (users' responsibility)
    Risk: MEDIUM (requires good UUID generation, documented)

recommendations for Fluxon v2.

================================================================================
KEY DESIGN DECISIONS & TRADE-OFFS
================================================================================

1. INTEGER CENTS EVERYWHERE
   Trade-off: More explicit (no surprise floats) vs slightly awkward API
   Example: amount_cents: 50000 instead of amount: 500.00
   Benefit: Eliminates rounding errors, precision issues

2. IMMUTABLE LEDGER ENTRIES
   Trade-off: Can't correct entries vs strong invariant
   Solution: Create reversal transaction (inverse ledger entries)
   Benefit: Audit trail is complete, correctable without data loss

3. DAILY RECONCILIATION (not real-time)
   Trade-off: Detects corruption 24h later vs low overhead
   Benefit: Simple, reliable, easy to audit
   Risk: Discrepancies undetected for hours (acceptable for most use cases)

4. AI-POWERED FRAUD (in addition to rules)
   Trade-off: Higher latency (API call) vs better detection
   Solution: Both rules (fast) AND AI (accurate)
   Benefit: Catches patterns humans miss, explainable

5. DB UNIQUE CONSTRAINT FOR IDEMPOTENCY (not app-level)
   Trade-off: Relies on DB feature vs no external dependencies
   Solution: Explicit UNIQUE constraint, assume error raised
   Benefit: Atomic, no race condition (if DB complies)
