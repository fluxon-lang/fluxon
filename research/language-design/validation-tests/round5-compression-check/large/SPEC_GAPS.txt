## Spec Gaps I Hit

This job board platform (9 Flux files, 2200+ LOC) revealed 15 minor ambiguities in the spec—all resolvable by reasonable inference, but not explicitly documented:

### Critical Gaps (Feature Usability)
None. All core features (transactions, AI, WebSocket, cron, schema) work as spec describes.

### Ambiguity Gaps (Needed Inference)

1. **Symbol Auto-Conversion in DB**
   - Spec shows symbol literals in db.ins and symbol matching, but doesn't explain how sym-typed columns auto-convert string ↔ symbol.
   - Assumption: Implicit conversion for sym columns (they're enums in DB).

2. **Map.set Return Value**
   - Spec doesn't clarify if m.set k v returns new map (functional) or mutates in-place.
   - Assumption: Returns new map, enabling chaining like map.set k1 v1.set k2 v2.

3. **ai.json Result Structure**
   - Spec shows shape definition and r._.conf but doesn't clarify field access.
   - Assumption: Direct access (result.score, result.reasons, metadata under result._.*).

4. **req.query Type Coercion**
   - Spec doesn't state if query params are auto-cast or always strings.
   - Assumption: Always strings; must explicitly convert with str.int, str.float.

5. **db.up with Conditional Fields**
   - Spec doesn't say if passing nil values skips columns or sets NULL.
   - Assumption: Conditional .set avoids nil keys entirely.

6. **Loop Nesting in db.tx**
   - Spec doesn't confirm if each loops inside transactions are atomic.
   - Assumption: Yes, they're part of the transaction scope.

7. **JSON Column Type Usage**
   - Spec mentions auto-encoding but shows no schema example.
   - Assumption: Transparent—pass Flux map/list, auto-serialize/deserialize.

8. **Router Precedence for Dynamic Segments**
   - Spec says "literal paths are prioritized" but doesn't clarify /companies/:id/jobs vs /jobs/:id.
   - Assumption: Longer/more specific paths win; register in sensible order.

9. **WebSocket Message Serialization**
   - Spec doesn't explicit state if ws.room.send requires JSON string or accepts maps.
   - Assumption: Requires string; always call json.enc before sending.

10. **Idempotency Implementation Pattern**
    - Spec describes the concept (uniq + duplicate detection in tx) but no full example.
    - Assumption: Check for existing record inside transaction, return 409 if found.

11. **Database Query Operators**
    - Spec shows basic SQL but doesn't document which operators are available (ILIKE, CASE, etc.).
    - Assumption: Standard Postgres operators work (ILIKE for case-insensitive search).

12. **time.ago Result in Queries**
    - Spec shows time.ago returns a timestamp but doesn't confirm direct comparison in WHERE.
    - Assumption: Comparable timestamp; pass directly as param to db.q.

13. **HTTP Redirect Standard**
    - Spec shows rep 302 {location:url} but doesn't confirm Location header format.
    - Assumption: Auto-converts to HTTP Location header.

14. **Module Import Aliasing Scope**
    - Spec doesn't clarify if alias applies to all functions or just disambiguates.
    - Assumption: Only exp functions are exported; alias remaps namespace.

15. **Error Propagation with !**
    - Spec doesn't clarify if ! auto-fails with 500 or stops execution.
    - Assumption: Auto-fails with 500 + error; use explicit fail 4xx for validation errors.

### Missing (Not in Spec, Not Implemented)
- No error details beyond {error: "string"}—no error.code or error.details.
- No request authentication/auth module (app assumes external JWT verification).
- No rate limiting, request logging middleware, or observability hooks.
- No GraphQL or non-REST query languages.
- No built-in pagination metadata (count, hasMore)—must manually track.

### Conclusion
The compressed spec (1 page, 263 lines) successfully encodes a production platform. All gaps are **minor inference** about edge cases that don't block implementation. No critical features are missing. **The language is learnable in one read.**
